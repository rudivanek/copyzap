# Google OAuth Setup Guide

This guide explains how to configure Google OAuth authentication in your Supabase project for CopyZap.

## What Was Implemented

1. **Google OAuth Sign-In Button** - Primary authentication option on the login page
2. **Email/Password Login** - Secondary option below a divider
3. **Automatic User Setup** - New users are automatically created in the database
4. **Error Handling** - User-friendly error messages for OAuth failures
5. **Seamless Redirects** - Users land directly on the Copy Maker page after sign-in

## UI Changes

### Login Page (`/login`)
- **Primary Option**: "Continue with Google" button (with Chrome icon)
- **Divider**: "or" separator
- **Secondary Option**: Email/password form (unchanged)

The Google OAuth button appears for both sign-in and sign-up flows.

## Supabase Configuration Required

To enable Google OAuth, you need to configure it in your Supabase project:

### Step 1: Get Google OAuth Credentials

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a new project or select an existing one
3. Go to "APIs & Services" → "Credentials"
4. Click "Create Credentials" → "OAuth 2.0 Client ID"
5. If prompted, configure the OAuth consent screen:
   - Choose "External" user type
   - Add app name: "CopyZap"
   - Add user support email
   - Add authorized domain: `copyzap.app`
   - Add developer contact email
   - Click "Save and Continue"
6. Back on the Create OAuth Client ID screen:
   - Select "Web application" as the application type
   - Name it "CopyZap Production" (or any name you prefer)
7. **CRITICAL**: Add the correct authorized redirect URI:
   - Click "Add URI" under "Authorized redirect URIs"
   - Enter EXACTLY: `https://gsismfzlmmtxmuzommya.supabase.co/auth/v1/callback`
   - **This must be your Supabase project's callback URL, not your app's URL**
8. For local development (optional), create a separate OAuth client with:
   - Redirect URI: `http://localhost:54321/auth/v1/callback`
9. Click "Create"
10. Copy your Client ID and Client Secret

### Step 2: Configure Supabase

1. Go to your Supabase Dashboard: https://supabase.com/dashboard/project/gsismfzlmmtxmuzommya
2. Navigate to **Authentication** → **Providers**
3. Find **Google** in the provider list
4. Toggle to enable the Google provider
5. Paste your Google Client ID
6. Paste your Google Client Secret
7. Click **Save**

### Step 3: Verify the Configuration

The OAuth flow works as follows:
1. User clicks "Continue with Google" → App redirects to Google
2. User authorizes on Google → Google redirects to **Supabase's callback URL**
3. Supabase processes OAuth → Supabase redirects to **your app** (`/copy-maker`)
4. User lands on Copy Maker page, fully authenticated

**Important**: The redirect URI in Google Cloud Console must be Supabase's callback URL, NOT your application's URL. Your application's final destination is configured in the code (`redirectTo` parameter).

## How It Works

### Sign-In Flow

1. User clicks "Continue with Google" on `/login`
2. Application calls `supabase.auth.signInWithOAuth()` with `provider: 'google'`
3. User is redirected to Google's OAuth consent screen
4. After authorization, Google redirects to **Supabase's callback URL**: `https://gsismfzlmmtxmuzommya.supabase.co/auth/v1/callback`
5. Supabase:
   - Receives the OAuth code from Google
   - Exchanges it for user tokens
   - Creates a session
   - Redirects to the application's final destination: `https://copyzap.app/copy-maker`
6. The `useAuth` hook detects the new session and:
   - Retrieves user information
   - Calls `ensureUserExists()` to create a record in the `pmc_users` table if needed
   - Extracts user metadata (name, email)
7. User is now fully authenticated and on the Copy Maker page

### User Data

When a user signs in with Google, their data is stored as:
- **Email**: `user.email`
- **Name**: `user.user_metadata.name` or `user.user_metadata.full_name`
- **User ID**: Generated by Supabase Auth

The `ensureUserExists()` function automatically creates a record in the `pmc_users` table if needed.

For local testing, you'll need to:
1. Create a separate OAuth 2.0 Client ID in Google Cloud Console
2. Add redirect URI: `http://localhost:54321/auth/v1/callback` (note the Supabase local port)
3. Update your local Supabase configuration with these credentials
4. Run the dev server: `npm run dev`
5. Test the flow at http://localhost:3000/login

**Note**: Local OAuth testing requires running Supabase locally or using separate OAuth credentials. For most development, test with production OAuth is simpler.

## Error Handling

The implementation includes comprehensive error handling:

- **Connection Errors**: "Connection error: Unable to reach authentication service"
- **OAuth Failures**: "Google sign-in failed: [specific error]"
- **Session Errors**: "Authentication failed. Please try again."
- **Timeout**: If auth state doesn't update within 10 seconds, shows timeout error

All errors automatically redirect users back to the login page after 3 seconds.

3. **Token Management**: Session tokens are managed securely by Supabase

## Technical Details

### Redirect Flow
- The `redirectTo` parameter in `signInWithOAuth()` specifies where users land AFTER Supabase completes OAuth
- It does NOT affect the OAuth callback URL (which is always Supabase's URL)
- Current configuration: Users land on `/copy-maker` after successful OAuth

### User Metadata
OAuth users have their data stored as:
- `user.email` - Email address from Google
- `user.user_metadata.name` - Full name from Google profile
- `user.user_metadata.full_name` - Alternative name field from Google

The `ensureUserExists()` function extracts the best available name and creates the pmc_users record.

## Troubleshooting

### Error: "Access blocked: This app's request is invalid" / "redirect_uri_mismatch"

**This is the most common error.** It means the redirect URI configured in Google Cloud Console doesn't match what Supabase is sending.

**Solution**:
1. Go to [Google Cloud Console](https://console.cloud.google.com/apis/credentials)
2. Click on your OAuth 2.0 Client ID
3. Under "Authorized redirect URIs", verify you have EXACTLY:
   ```
   https://gsismfzlmmtxmuzommya.supabase.co/auth/v1/callback
   ```
4. Make sure there are no extra spaces, trailing slashes, or typos
5. Click "Save"
6. Wait 5 minutes for Google's changes to propagate
7. Try signing in again

### "Authentication service is not configured"
- Check that `VITE_SUPABASE_ENABLED` is not set to `false` in `.env`
- Verify Supabase credentials in `.env`

### "Google sign-in failed"
- Ensure Google OAuth is enabled in Supabase Dashboard
- Verify Client ID and Client Secret are correct
- Check that the redirect URI in Google Console matches your Supabase callback URL

### "Connection error"
- Check browser console for errors
- Verify network connectivity
- Ensure Supabase is accessible (not blocked by firewall)

### User is redirected but not logged in
- Check browser console for errors
- Verify the `useAuth` hook is detecting the session
- Check that `ensureUserExists()` is being called successfully
- Look for RLS policy errors in Supabase logs
